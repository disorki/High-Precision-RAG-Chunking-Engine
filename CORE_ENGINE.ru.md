# Документация Ядра RAG (Core Engine)

Данный документ содержит глубокое техническое описание работы "движка" системы. Предназначен для DevOps-инженеров и системных архитекторов.

## 1. Архитектура системы

Система построена по классической схеме **Retrieval-Augmented Generation (RAG)** с локальным исполнением всех компонентов.

### Ключевые компоненты:
1.  **FastAPI Backend**: Координатор всех процессов, управление бизнес-логикой и API.
2.  **RAG Pipeline**: Сервис извлечения текста, чанкинга и генерации эмбеддингов.
3.  **Vector Database (PostgreSQL + pgvector)**: Хранение и семантический поиск чанков.
4.  **Ollama**: Локальный LLM-сервер для генерации эмбеддингов и ответов.
5.  **Reranker (Cross-Encoder)**: Опциональный слой уточнения результатов поиска.

---

## 2. Пайплайн обработки документов (`rag_pipeline.py`)

Сервис отвечает за превращение сырых файлов в структурированные векторные данные.

### Этапы обработки:
1.  **Извлечение текста (Extraction)**:
    *   **PDF**: Используется `pypdf`. Включает поддержку многостраничности.
    *   **DOCX**: Парсинг абзацев и таблиц. Таблицы конвертируются в Markdown-подобный текст для сохранения структуры.
    *   **XLSX**: Каждый лист обрабатывается как отдельная таблица.
    *   **TXT**: Прямое чтение с автоопределением кодировки (UTF-8, CP1251).
2.  **Чанкинг (Chunking)**:
    *   Используется `RecursiveCharacterTextSplitter`.
    *   **Размер (chunk_size)**: 1500 символов (настраиваемо).
    *   **Перекрытие (overlap)**: 300 символов для сохранения контекста между блоками.
    *   **Разделители**: `\n\n`, `\n`, `. `, ` ` (в порядке убывания приоритета).
3.  **Генерация эмбеддингов (Embeddings)**:
    *   Модель по умолчанию: `nomic-embed-text` (через Ollama).
    *   Параллельная обработка чанков (настраиваемая конкурентность через `embedding_concurrency`).
    *   Механизм повторов (retry) при сбоях Ollama.

## 3. Поиск и извлечение (`retrieval.py` & `reranker.py`)

Сервисы обеспечивают нахождение наиболее релевантных данных для вопроса пользователя.

### Процесс поиска:
1.  **Векторный поиск**:
    *   Запрос пользователя превращается в эмбеддинг.
    *   Выполняется поиск в PostgreSQL с использованием косинусного расстояния (`<=>`).
    *   **Порог (Similarity Threshold)**: По умолчанию 0.05.
2.  **Реранжирование (Reranking)**:
    *   Если включен (`enable_reranker=True`), используется модель Сross-Encoder (например, `ms-marco-MiniLM-L-6-v2`).
    *   Cross-Encoder оценивает реальную релевантность пары "Вопрос-Чанк", что значительно точнее обычного векторного поиска.
3.  **Сборка контекста**:
    *   Топ-K чанков (по умолчанию 20) объединяются в единую строку.
    *   Добавляются метаданные источников (название файла, номер страницы).
    *   Лимит контекста: 16000 токенов.

## 4. Конфигурация (DevOps Manual)

### Переменные окружения (.env)
| Переменная | Описание | Значение по умолчанию |
| :--- | :--- | :--- |
| `DATABASE_URL` | Подключение к PostgreSQL | `postgresql://...` |
| `OLLAMA_BASE_URL` | Адрес сервера Ollama | `http://localhost:11434` |
| `EMBEDDING_MODEL` | Модель для векторов | `nomic-embed-text` |
| `CHAT_MODEL` | Модель для генерации (RAG) | `qwen2.5:7b` |
| `AGENT_MODEL` | Модель для Агента | `qwen2.5:14b` |
| `CHUNK_SIZE` | Размер блока текста | `1500` |
| `TOP_K_CHUNKS` | Кол-во чанков в контексте | `20` |
| `ENABLE_RERANKER`| Использовать Cross-Encoder | `false` |

### Рекомендации по развертыванию:
1.  **GPU Acceleration**: Для работы Ollama крайне желательно наличие NVIDIA GPU. В Docker-compose должен быть проброшен `deploy.resources.reservations.devices`.
2.  **Оптимизация БД**: Убедитесь, что расширение `vector` установлено в PostgreSQL. Для больших объемов данных (100к+ чанков) рекомендуется создать индекс `HNSW` или `IVFFlat`.
3.  **Масштабирование**: Бэкенд является stateless. Можно запускать несколько реплик за балансировщиком.

## 5. Интеграция с Яндекс.Диском (`yandex_disk.py`)

*   Работает через **OAuth 2.0**.
*   **Синхронизация**: Сравниваются хеши локальных файлов и файлов в облаке.
*   **Автоматизация**: Реализован механизм фоновой проверки обновлений (интервал настраивается).
*   **Импорт**: Файлы скачиваются, регистрируются в БД и автоматически отправляются в RAG-пайплайн.

## 6. Работа Агента (`agent_chat.py`)

Агент использует подход **Native RAG**:
1.  **Анализ вопроса**: LLM определяет, какие данные нужны (глобальный поиск или по документу).
2.  **Reasoning**: Формирует внутренний план поиска.
3.  **Synthesis**: Получает чанки, анализирует их и формирует ответ со строгими ссылками на страницы (`[Стр. X]`).
4.  **Safety**: Если данных нет, агент обязан сообщить об этом, а не галлюцинировать.
